Введение

Актуальность. Совместное прослушивание музыки и коллективное кураторство
плейлистов сохраняют социальную значимость в цифровую эпоху. Современные
стриминговые сервисы ориентированы на индивидуальное потребление и
алгоритмические рекомендации, что приводит к фрагментации опыта пользователей
и снижению возможностей для живого коллективного взаимодействия. OmniPlayer —
веб-приложение, направленное на восстановление социального контекста прослушивания:
загрузка собственной музыки, тематические комнаты с синхронным воспроизведением
и встроенный текстовый чат.

Цель работы

Разработка веб-приложения "OmniPlayer", обеспечивающего:

- хранение и воспроизведение пользовательского аудиоконтента;
- создание и управление комнатами для синхронного прослушивания;
- синхронизацию проигрывания и управление общей очередью треков;
- встроенную коммуникацию (чат) и базовые механизмы модерации.

Задачи

1. Проанализировать предметную область и существующие аналоги.
2. Описать и формализовать требования (функциональные и нефункциональные).
3. Спроектировать архитектуру и базовую структуру данных.
4. Реализовать backend- и frontend-компоненты и механизм синхронизации.
5. Провести тестирование и подготовить руководство пользователя и материалы для сдачи.

Объект исследования

Процесс организации совместного прослушивания музыки в веб-среде и
инструменты для его поддержки (синхронизация, очереди, чат).

Предмет исследования

Веб-приложение OmniPlayer и реализованные в нём механизмы: загрузка треков,
структура комнат, очередь треков, синхронизация воспроизведения и чат.

Методы исследования

Аналитические (сравнение аналогов), проективные (UML/ERD), прикладные
(реализация прототипа), экспериментальные (тестирование синхронизации),
а также методы сбора и анализа пользовательской обратной связи (при наличии).

1. Теоретическая часть

1.1 Анализ предметной области

Область — социальное прослушивание музыки в реальном времени. Типовой сценарий:
регистрация, создание/поиск комнаты, добавление треков в очередь, синхронное
прослушивание, текстовая коммуникация. Выявленные проблемы: зависимость от
платных подписок у коммерческих сервисов, отсутствие встроенного чата в
многих стриминговых решениях, нестабильность внешних бот-решений (Discord).

1.1.1 Сравнительный анализ аналогов

Таблица 1 — сравнение ключевых платформ по релевантным признакам.

| Платформа               | Загрузка личной музыки |        Комнаты | Синхронность | Встроенный чат | Ограничения                                                                       |
| -------------------------------- | -----------------------------------------: | --------------------: | -----------------------: | --------------------------: | -------------------------------------------------------------------------------------------- |
| OmniPlayer (наш проект) |                                       Да | Да (комнаты) |         Да (WebSocket) |                        Да | Самостоятелен, поддержка локальных загрузок           |
| Spotify (Jam)                    |                    Нет (каталог) |               Огр. |       Частичная |                      Нет | Требует подписки; не поддерживает локальные файлы |
| Discord + боты               |                                     Нет |                  Да |          Да (боты) |                        Да | Зависимость от третьих ботов и API, нестабильно         |

Вывод: ни один из популярных сервисов не покрывает одновременно загрузку
локальной музыки, независимые комнаты, встроенный чат и отсутствие привязки к
платным подпискам — это обосновывает разработку собственного решения.

1.2 Варианты решения и обоснование выбора

Рассмотрены три подхода: интеграция со сторонним стримингом, использование
коммуникационных платформ (Discord) и собственное веб‑приложение. Для целей
курсовой работы выбран последний — даёт полный контроль над функционалом и
позволяет реализовать требования учебной задачи.

1.3 Требования к системе

Функциональные требования (сокращённо):

- Регистрация и авторизация пользователей (JWT).
- Создание комнат: публичные и с паролем; просмотр списка активных комнат.
- Управление очередью треков: добавление, удаление, перемещение, поиск треков
  через `yt-dlp` (SoundCloud/YouTube).
- Синхронное воспроизведение: передача команд play/pause/seek и смещение
  времени не более 1 с для слушателей.
- Текстовый чат в комнате с историей и модерацией (удаление сообщений, блокировка пользователей).
- Личная библиотека и плейлисты пользователя.

Нефункциональные требования:

- Производительность: ответы API для основных операций < 500 ms.
- Безопасность: хранение паролей с bcrypt, access/refresh JWT, разграничение прав.
- Надёжность: транзакционная работа с очередью и атомарные операции над данными.
- Кросс-браузерность и адаптивный интерфейс.
- Контейнеризация и воспроизводимость окружения (Docker Compose).

2. Технологическая часть

2.1 Выбор инструментов

Решение реализовано как распределённый монолит (frontend + backend отдельно).
Выбранный стек:

- Backend: FastAPI (Python) — асинхронность, Pydantic для валидации, OpenAPI.
- WebSocket: встроенные возможности Starlette/FastAPI для синхронизации.
- DB/ORM: PostgreSQL + SQLAlchemy; Alembic для миграций.
- Frontend: React 18 + Vite, TailwindCSS для стилей, Recharts для графиков.
- Инфраструктура: Docker, docker-compose.

2.2 Архитектура и проектирование

Архитектура: клиент (SPA) ↔ REST API (FastAPI) + WebSocket gateway ↔ БД.
Lite‑версия движка воспроизведения на сервере синхронизирует состояние комнаты
и рассылает события через WebSocket. Сервисная прослойка (Service Layer)
нижает связность между роутерами и бизнес-логикой.

Плейсхолдеры для диаграмм (создать в `docs/images/`):

- `docs/images/architecture.png` — архитектурная диаграмма.
- `docs/images/usecase.png` — UseCase диаграмма (актёры: гость, пользователь, админ).
- `docs/images/erd.png` — ERD базы данных (таблицы users, rooms, queue_items, tracks, messages).
- `docs/images/class_diagram.png` — диаграмма классов/сервисов.

2.3 Структура базы данных (кратко)

Ключевые сущности:

- `users` — пользователи (id, email, username, hashed_password, role).
- `rooms` — комнаты (id, name, owner_id, password_hash, created_at, state).
- `queue_items` — элементы очереди (id, room_id, track_id, position, added_by).
- `tracks` — метаданные треков и ссылки на asset (id, title, source, duration, asset_path).
- `messages` — чат (id, room_id, user_id, text, created_at).

ERD нужно выгрузить в `docs/images/erd.png` и приложить к отчёту.

3. Практическая часть

3.1 Реализация компонентов (обзор)

Проект в репозитории содержит backend-часть в `backend/app/`.

Backend (основные модули и файлы):

- `app/main.py` — инициализация FastAPI, подключение роутеров и middleware.
- `app/room/queue.py` — логика управления очередью (CRUD, reorder).
- `app/room/manager.py` и `app/room/room_state.py` — управление состоянием комнаты.
- `app/realtime/room_gateway.py` — WebSocket gateway для синхронизации и чата.
- `app/player/controller.py` — операции play/pause/seek, start_playback.
- `app/storage.py` / `app/core/config.py` — загрузка и хранение ассетов.

Frontend: SPA (папка `frontend/`) реализует страницы для входа, списка комнат,
интерфейса комнаты (плеер + чат + очередь). WebSocket клиент получает события
подключения, изменения очереди и состояния воспроизведения.

Основные реализованные сценарии:

- Создание комнаты с опциональным паролем.
- Добавление треков в очередь (поиск через интеграцию yt-dlp или загрузка файла).
- Запуск воспроизведения администратором комнаты; распространение состояния
  (now_playing, position, paused) слушателям через WebSocket.
- Чат в реальном времени с модерацией (удаление сообщения/бан).
- Персистентная очередь и история сообщений в БД.

3.2 Тестирование

Покрытие тестами (пример):

- unit-тесты для `queue` (добавление/удаление/перестановка).
- интеграционные тесты для REST-эндпоинтов авторизации и операций с комнатами.
- ручное тестирование синхронизации: несколько браузеров подключаются в комнату —
  проверяется соблюдение задержки синхронизации < 1 с при нормальной сети.

Тестовые артефакты: `tests/test_queue.py`, `tests/test_rooms.py` (плейсхолдеры,
при отсутствии — создать и заполнить).

3.3 Ввод в эксплуатацию

Рекомендации для развёртывания в проде:

1. Подготовить `.env` с `DATABASE_URL`, `JWT_SECRET`, `ALLOWED_HOSTS`.
2. Использовать PostgreSQL, NGiNX как reverse-proxy, SSL/TLS.
3. Запуск через `docker compose -f docker-compose.prod.yml up -d`.
4. Настроить бекапы БД и мониторинг логов (ELK/Loki).

Команды для локальной проверки приведены в разделе запуска (ниже).

Запуск проекта (локально, dev)

```bash
# Backend (виртуальное окружение)
python3 -m venv venv
source venv/bin/activate
pip install -r backend/requirements.txt
cd backend
alembic upgrade head
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# Frontend
cd ../frontend
npm install
npm run dev

# Альтернативно — docker-compose
docker compose up --build
```

Документация API доступна по `/docs` после запуска backend.

организации совместного прослушивания музыки с поддержкой загрузки личных

- улучшение точности синхронизации (плавающая калибровка пинга);

- интеграция соц-функций: профили, рекомендации и механизмы поиска единомышленников.
